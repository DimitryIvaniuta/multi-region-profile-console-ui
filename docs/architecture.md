# Frontend Architecture

## Shell

The application uses a banking-style shell:

- left sidebar navigation;
- top region/status bar;
- central workspace;
- footer with operational notes;
- skip link and keyboard-visible focus states.

Routing is hash-based and intentionally small to avoid an extra runtime router dependency. The routes are:

- `#/dashboard`
- `#/create`
- `#/update`
- `#/read`
- `#/compare`
- `#/ops`
- `#/audit`
- `#/settings`

## API client

The API client is a thin `fetch` wrapper with:

- timeout via `AbortController`;
- per-request `X-Correlation-Id`;
- `X-API-Key` and `X-Internal-Key` header modes;
- JSON error parsing compatible with backend `ErrorResponse`;
- `credentials: 'omit'` to avoid accidental cookie-based auth coupling;
- retry only for idempotent reads and transient HTTP/network failures.

## Runtime configuration

The console supports two configuration paths:

1. build-time Vite variables for local development;
2. runtime `env.js` generated by Docker/nginx entrypoint for immutable image promotion.

`sessionStorage` overrides are still available from the **Connection Settings** page for local operator workflows.

## Forms

Each backend capability has a dedicated controlled form:

- profile creation;
- profile update;
- profile deactivation;
- regional profile read with optional `minVersion`;
- all-region profile comparison;
- operations refresh;
- bounded outbox replay;
- connection health check.

## State

The console uses local React state because workflows are operator-driven and low-volume. This avoids extra query/state dependencies and keeps the supply-chain surface small.

The activity audit is browser-session-local and non-authoritative. Production compliance audit should remain server-side.
